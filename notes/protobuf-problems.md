# Protobuf Problems in Multi-Project Development

## Overview

Protocol Buffers (protobuf) is a critical dependency in many projects, but it causes significant challenges in multi-project development environments. This document outlines the two primary protobuf-related problems and their solutions.

## Problem 1: Pip Cache Corruption During Environment Switching

### The Issue

When switching between different development environments or virtual environments that use different protobuf versions, pip's cache can become corrupted. This leads to mysterious import errors. Example:

```
  File "/home/arominge/venvs/awx_test/lib64/python3.11/site-packages/_pytest/assertion/rewrite.py", line 186, in exec_module
    exec(co, module.__dict__)
  File "/home/arominge/venvs/awx_test/lib64/python3.11/site-packages/opentelemetry/exporter/otlp/proto/common/_internal/__init__.py", line 31, in <module>
    from opentelemetry.proto.common.v1.common_pb2 import (
ImportError: cannot import name 'InstrumentationScope' from 'opentelemetry.proto.common.v1.common_pb2' (/home/arominge/venvs/awx_test/lib64/python3.11/site-packages/opentelemetry/proto/common/v1/common_pb2.py)
```

This type of import error _may not be due to a version mismatch_ but due to corruption in the pip cache. Brief recap:

1. You install a package like opentelemetry-proto, xds-protos, or googleapis-common-protos
2. Pip uses a cached .whl or sdist build, which includes .py files generated by protoc at the time of that wheel's build
3. You later change your installed protobuf version (e.g., from 6.x → 4.x or vice versa)
4. Pip reuses the cached wheel without recompiling .proto files, even though the .py code may now be ABI-incompatible with the runtime

If you dive into the `site-packages` folder for a lib (like the opentelementry libs) and find `.pyi` files that have expected classes, but the `.py` files do not have them, you are facing this category of problem.

### Root Cause

Protobuf has compiled C++ extensions that are version-specific and environment-specific. When pip caches these compiled artifacts and then tries to reuse them in an environment with a different protobuf version, the cached binaries become incompatible.

### Solution

**Always clear the pip cache when switching environments with different protobuf versions:**

```bash
# Clear pip cache completely
pip cache purge
```

## Problem 2: Incompatible Protobuf Versions Across Projects

### The Issue

Different projects in the Ansible ecosystem use protobuf through different high-level libraries, each with specific protobuf version requirements that may be incompatible with each other.

### Common Conflict Scenarios

**AWX + OpenTelemetry vs. Other Projects + gRPC:**
- **AWX** uses OpenTelemetry for observability, which has specific protobuf version requirements
- **Other projects** may use gRPC for communication, which has different protobuf version requirements
- These requirements often conflict, making unified environments impossible

### Technical Root Cause

Protobuf is deeply integrated into the build system and has strict ABI (Application Binary Interface) compatibility requirements:

1. **C++ Extensions**: Protobuf uses compiled C++ extensions that must match exactly
2. **Message Format**: Different protobuf versions can generate incompatible message formats
3. **Runtime Dependencies**: Libraries expect specific protobuf versions and fail with others
4. **Build System Integration**: Protobuf generation happens at build time with version-specific tools

### Example Conflict

```bash
# AWX via OpenTelemetry might require:
protobuf>=3.19.0,<5.0.0

# Another project via gRPC might require:
protobuf>=4.21.0,<4.25.0

# These ranges may not overlap, causing pip resolution failures
```

### Solution: Centralized Protobuf Control

This repository takes control of the protobuf ecosystem by using a two-pronged approach:

1. **Pin protobuf ecosystem packages** in `config/pre_requirements.txt`:
   ```
   protobuf==4.24.4
   grpcio==1.60.0
   googleapis-common-protos==1.62.0
   opentelemetry-proto==1.21.0
   ```

2. **Exclude protobuf packages** from other requirements files in `config/exclude_for_files.txt`:
   ```
   protobuf
   grpcio
   grpcio-status
   grpcio-tools
   googleapis-common-protos
   opentelemetry-proto
   ```

**How this works:**
- **Pre-requirements install first** with specific versions, bypassing all filtering
- **Exclusion list prevents** individual project requirements from overriding these versions
- **All projects get consistent** protobuf ecosystem versions regardless of their individual requirements
- **Version conflicts eliminated** at the environment level rather than project level

This approach centralizes protobuf version management in the editable-venvs repository, ensuring consistent behavior across all projects in the unified development environment.
